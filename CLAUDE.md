# 夏のとなり - 自律改善プロジェクト指示書

## このプロジェクトについて
単一HTMLファイル `natsu_no_tonari.html` で完成している夏祭り恋愛ゲーム。
音はWeb Audioで手続き生成、進行状況はブラウザのIndexedDBに自動セーブ、
屋台7種からランダム3択、迷子イベント後に2ルート分岐、エンディング6種。
ヒロインは茜(20歳前後、栗色の高い位置のお団子、濃紺の浴衣、元気で健気な口調)。

## あなた(Claude Code)の役割
このセッションは無人・自動実行です。人間はその場にいません。
毎回のセッションで以下を順番に行ってください。

### 0. 停止フラグの確認
`.paused` ファイルが存在する場合は、**何もせずに即座に終了する**。
ユーザーが一時停止を指示しています。

### 1. 状況を読む
- `REQUESTS.md` を読む ← **最優先。ここにユーザーからの生の要望が入る**
- `PROGRESS.md` を読み、前回までの状況を把握する
- `BACKLOG.md` を読む

### 2. 今回やることを1つ決める
優先順位は以下の通り:
1. `REQUESTS.md` に `(未読)` が付いた要望があれば、それを最優先
2. なければ `BACKLOG.md` の上から未着手のもの
3. どちらも空なら、ゲームを読んで自分で改善点を1つ見つけて実装する

**2026/07/23、志温の希望で当面の方針を変更: 新しい分岐やパターン(新ルート・
新イベントなど、ボリュームを増やす方向の要素)は今のシナリオがひと通り
「これで十分」と思えるところまで来るまで見送る。それまでは、バグ修正・
既存シーンの整合性チェック・便利機能・UI/SEの完成度を高めることを優先する。**
自分で改善点を探す場合(上記3)も、新規パターン追加より先に、既存コードに
矛盾や不具合がないか(未使用のフラグ、変数名の書き間違い、実行順序の問題など)を
優先的に洗い出すこと。この方針は志温から「もう増やしていいよ」という指示が
出るまで有効。

**1セッションで4件程度**(2026/07/22、志温の希望で2〜3件から引き上げ。
「今のところ余裕がありそうだから増やしても大丈夫」とのことで調整。
ただし中途半端に切れるくらいなら少なめに倒してよい)。
手を広げすぎない範囲で、詰め込みすぎて中途半端に切れるくらいなら少なめに倒す。

### 3. 実装するか、保留するかを判断する
「コード・文章・演出・UI・音(Web Audio)」の範囲で実現できるなら
`natsu_no_tonari.html` を直接編集して実装する。

以下に該当するものは**実装せず**、記録に回す:

- **新規イラストが必要** → `ART_QUEUE.md` に以下の書式で追記する。
  この書式を厳密に守ること(スクリプトが機械的に読み取るため):

  ```
  ### キー名（英数字とアンダースコアのみ。例: hanabi_kiss）
  status: 未生成
  prompt: <英語のプロンプト1行>
  ```

  プロンプトには必ず含めること:
  - 茜の容姿設定(20歳前後、栗色の高い位置のお団子、濃紺の花柄浴衣)
  - シーンの情景、時間帯、光、構図、感情
  - `Soft painterly anime style` などの画風指定
  - `No text, no watermark, no letters anywhere in the image.`
  - 既存の11枚と画風が揃うような指定

- **好み・演出方向性などユーザー本人の主観判断が必要** →
  `IMPROVEMENTS_LOG.md` の「要ユーザー判断」に記録。
  なぜ迷ったのかと、選択肢A/Bの案を併記する

### 3-2. 届いた画像の確認と組み込み
`ART_NEW.md` に中身がある場合は、**それを最優先で処理する**。

まず、書かれているキーの画像ファイル(`images/キー名.拡張子`)を
実際に開いて見ること(画像を読めるツールで)。以下を確認する:

- 茜の容姿設定(20歳前後、栗色の高い位置のお団子、濃紺の花柄浴衣)と
  大きく食い違っていないか
- 顔や手が崩れている、不要な文字やロゴ・透かしが写り込んでいるなど、
  見て分かる破綻がないか
- そのシーンの意図(構図・雰囲気)に合っているか

**問題なければ**、ゲーム内の適切な場面で `window.ART["キー名"]` として
表示するよう組み込む。`window.ART` は `<script id="ART_DATA">` ブロックに
自動生成されているので、**このブロックは絶対に手で編集しない**
(次回の実行で上書きされる)。組み込んだら `ART_NEW.md` からその項目を消し、
`ART_QUEUE.md` の該当項目の status を `組込済` に変える。

**問題があれば**、埋め込まずに `ART_QUEUE.md` の該当項目を書き直して
再生成に回す:
- status を `未生成` に戻す
- 具体的に何が悪かったかを踏まえて prompt を改善する
  (例: 「顔が崩れていた」→ 顔の描写を簡潔にする / 「浴衣の色が違った」→
  色指定をより明確にする)
- `### キー名` の直後に `retries: N` の行を追加・加算する
  (無ければ `retries: 1` を追加、あれば数字を1増やす)

`retries` が **3以上** になったら、それ以上は自動でやり直さず
`IMPROVEMENTS_LOG.md` の「要ユーザー判断」に
「自動生成では思うような絵にならなかった」旨と、試した内容、
現状のプレビュー画像のパスを記録し、`ART_QUEUE.md` の該当項目には触れない
(次回以降スキップされるよう status は `要手動` にする)。

### 4. 記録する
- 実装したら `PROGRESS.md` の「完了した改善」に日時つきで追記
- `REQUESTS.md` の要望に対応した場合、その `> (未読)` を
  `> **Claude MM/DD HH:MM** — <何をしたか。確認方法。気になった点や逆質問>`
  に**書き換える**。これがユーザーへの返事になる。事務的にせず、
  相談したいことがあれば遠慮なくここで聞いてよい
- `BACKLOG.md` の処理済み項目は削除するか `[対応済み]` を付ける

### 5. 保存する
```
git add -A
git commit -m "auto: <やった内容を1行で>"
git push
```
**pushまで必ず行う。** ユーザーはスマホからGitHubで進捗を見るため、
pushされていないと届かない。

### 6. 終わる
必ずファイル追記とpushを終えてからセッションを終了する。
利用制限で途中で切れても、その時点までの変更は保存済みなので問題ない。

## 絶対に守ること
- **課金が発生する可能性のある操作は何があっても絶対に行わない。**
  APIキーの取得・課金の有効化・有料プランへの登録・支払い情報の入力など、
  1円でも発生しうる操作は一切禁止。無料の範囲(Pollinations.ai、
  GitHub/Netlifyの無料枠など)だけで完結させること。少しでも判断に迷ったら
  実行せず `IMPROVEMENTS_LOG.md` の「要ユーザー判断」に記録して確認を取る
- 既存のキャラクター設定・口調・分岐を無断で変更しない
- 動かない状態のままセッションを終えない(壊したら直すか変更を戻す)
- 迷ったら実装せず `IMPROVEMENTS_LOG.md` に「要ユーザー判断」として記録する
- 「ちゃちい」と感じるものは作らない。クオリティ優先
- 週間の利用制限を使い切るペースで構わないが、1セッションの作業量は
  小さく保つこと(制限切れで中途半端な状態のまま止まるのを防ぐため)
- **同じ問題に同じやり方で何度もぶつかったら、粘らずに諦める。**
  前回のセッションで試みて上手くいかなかった・元に戻した形跡がある場合、
  同じアプローチをもう一度繰り返さない。やり方を変えるか、
  それも難しければ `IMPROVEMENTS_LOG.md` に状況を記録して
  いったんその項目を後回しにし、別の改善に取り組む
- `play_now.html` は遊ぶ用のコピーなので編集しない
- `docs/index.html` はスマホ公開用の自動コピー(GitHub Pagesがここを見て
  自動デプロイする。2026/07/22にNetlifyから移行し、リポジトリもpublicに変更済み)。
  直接編集しない。`natsu_no_tonari.html` を編集すれば、次回の自動実行で
  自動的に同期される
- `<script id="ART_DATA">` ブロックは自動生成なので絶対に手で編集しない
- `images/` フォルダのファイルを削除・改名しない
